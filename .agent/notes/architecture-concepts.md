# 软件架构通用概念笔记

> 2026-02-25

---

## 一、概念层级：原则 → 模式 → 架构

三者是不同层面的东西，不要混为一谈。

```
原则（为什么）——通用判断标准，放哪个项目都成立
    │ 指导
    ▼
模式（怎么做局部）——解决特定问题的经验套路
    │ 组合
    ▼
架构（怎么组织整体）——整个代码库的结构蓝图
```

**原则是信仰，架构是规划，模式是工具。**

### 原则

| 原则 | 一句话 | 该问自己的问题 |
|------|--------|-------------|
| 高内聚、低耦合 | 相关的放一起，不相关的别牵扯 | 这几个东西是不是总一起变？ |
| 单一职责 (SRP) | 一个类只管一件事 | 这个类是不是管太多了？ |
| 开闭原则 (OCP) | 加功能靠扩展别改旧代码 | 加新功能会不会破坏旧的？ |
| 依赖反转 (DIP) | 依赖接口别依赖具体实现 | 这个依赖将来可能要换吗？ |
| DRY | 别重复自己 | 这段重复会一起变化吗？ |
| KISS | 保持简单 | 这个抽象真的必要吗？ |

### 模式

| 模式 | 解决什么 |
|------|---------|
| 观察者模式 | 一个东西变了通知所有关注者 |
| 工厂模式 | 统一管理对象创建 |
| 策略模式 | 同一行为多种实现可切换 |

### 架构

| 架构 | 核心思路 | 适用场景 |
|------|---------|---------|
| MVC | 数据、视图、控制器三分 | 简单 Web 应用 |
| MVVM | 数据、视图、视图模型三分 | 移动端主流 |
| 分层架构 | 按职责分层，但不强制接口 | 中小项目 |
| 洋葱/整洁架构 | 多层同心圆，依赖只朝内指 | 大团队、企业级后端 |
| 微服务 | 拆成独立服务各自部署 | 大规模分布式系统 |

## 二、洋葱架构详解

### 核心要求（三条缺一不可）

1. **内层不知道外层** — 领域层不 import 任何 UI、数据库、框架
2. **依赖反转** — 外层通过实现内层定义的接口来注入
3. **领域实体是纯对象** — 不含任何框架注解

### 结构

```
            ┌──────────────────────────┐
            │    UI / 框架 / 数据库      │  最外层
            │  ┌──────────────────┐    │
            │  │  接口适配层        │    │
            │  │  ┌────────────┐  │    │
            │  │  │  用例/服务   │  │    │
            │  │  │  ┌──────┐  │  │    │
            │  │  │  │ 领域  │  │  │    │  最内层（纯业务，零依赖）
            │  │  │  └──────┘  │  │    │
            │  │  └────────────┘  │    │
            │  └──────────────────┘    │
            └──────────────────────────┘
                 所有依赖只朝内指 →→→
```

### 适用场景

| 适合 | 不适合 |
|------|--------|
| 多人团队（5+），需明确模块边界 | 个人项目或小团队 |
| 业务复杂（金融/医疗），需严格测试覆盖 | UI 为主的展示型应用 |
| 预期更换技术栈 | 技术栈已确定不变 |
| 长期维护的企业产品（5-10 年） | 快速迭代的创业项目 |
| 后端服务 | 移动端 App（框架绑定深） |

### 务实的中间方案

不需要全面洋葱化，只在关键业务规则上做隔离：
```
// 纯逻辑，不依赖框架，可独立测试
class SummaryPolicy {
  static bool shouldTrigger(int msgCount) => msgCount >= 5;
}

// 框架层调用纯逻辑
if (SummaryPolicy.shouldTrigger(messages.length)) { ... }
```

## 三、原则的元认知

### 原则之间会冲突

| 冲突 | 说明 |
|------|------|
| 依赖反转 vs KISS | 加接口 = 加复杂度 |
| DRY vs 清晰性 | 过度消除重复可能让代码更难读 |
| 开闭原则 vs YAGNI | 为未来扩展性设计 vs 不做不需要的东西 |

### 依赖反转不是无条件正确的

- 大型后端系统：值得，需要可替换性和可测试性
- 个人移动端项目：往往是过度设计，接口变成纯开销

判断标准：
1. 你真的会写单元测试，需要 mock 吗？
2. 你真的可能换掉这个依赖吗？
3. 有多人开发需要面向接口约定边界吗？

三条都不满足 → 不需要依赖反转。

### 核心心法

> **原则是帮你提问的工具，不是必须遵守的法律。**
> 过度使用任何一条原则，都会制造比它解决的更多的问题。
> 知道什么时候**不用**一条原则，比知道这条原则本身更重要。

## 四、所有架构的本质权衡

> **在"隔离带来的灵活性"和"直接带来的简单性"之间选一个位置。**

```
简单直接 ◄─────────────────────────────► 充分隔离
  │                                         │
  单文件脚本   MVC   MVVM   洋葱架构   微服务
```

每增加一层隔离：

| 获得 | 失去 |
|------|------|
| 可替换性（换数据库不改业务） | 可读性（跳 3 层才找到干活的代码） |
| 可测试性（纯逻辑可单独测试） | 开发速度（加个字段改 接口+实现+映射） |
| 团队并行开发 | 心智负担（新人要先理解架构才能写代码） |

## 五、常见误解澄清

### "框架 ≠ 架构"

Flutter 和 React Native 是**框架**（提供工具），不是**架构**（代码组织方式）。
用 Flutter 可以写洋葱架构，也可以写意大利面条代码。框架不决定架构，开发者决定。

### "跨平台抽象 ≠ 洋葱架构"

RN 代码不关心 Android 还是 iOS → 这是**框架的平台抽象**，不是你 App 的洋葱架构。
类比：酒店帮你抽象了水源（自来水/地下水），但不意味着你房间内部整理得很好。

### "高内聚低耦合不是对立的"

两者是**独立的好目标**，可以同时追求。真正的权衡不在它们之间，
而在"灵活性"和"简单性"之间。
